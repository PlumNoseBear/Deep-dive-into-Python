# «Углубленный Python» от VK Education.
Первое ДЗ
Ты разрабатываешь сервис-агрегатор, который периодически парсит статьи с Хабра. 
Внешние сервисы нестабильны: они могут отвечать долго, возвращать ошибки сети или временно быть недоступными.
Если сервис "упал", нет смысла продолжать слать запросы (это может усугубить ситуацию или быть расценено как DDoS). Необходимо реализовать механизм защиты, который отслеживает здоровье внешнего сервиса и принимает решение: выполнить запрос, подождать или сразу вернуть ошибку.
Твоя задача - реализовать декоратор, реализующий упрощенную версию паттерна Circuit Breaker.
Circuit Breaker – это архитектурный паттерн, предназначенный для повышения отказоустойчивости распределенных систем путем предотвращения каскадных сбоев при недоступности внешних сервисов. 
Он работает по аналогии с электрическим автоматом: если количество ошибок при вызове удаленного ресурса превышает заданный порог, цепь “размыкается”, и все последующие запросы блокируются мгновенно, без реальной попытки обращения к проблемному сервису. 
Это дает системе время на восстановление (охлаждение), экономит ресурсы и защищает от перегрузки, а через определенный интервал выключатель переходит в пробный режим, чтобы проверить, возобновилась ли работа сервиса.

Требования к коду
 — Декоратор должен быть параметризируемым (декоратор является фабрикой декораторов).
 — Состояние сервиса должно храниться в замыкании декоратора (не глобально, а привязано к конкретной декорируемой функции).
 — Необходимо использовать стандартную библиотеку.
 — Код должен соответствовать PEP 8.

Декоратор должен принимать следующие аргументы:
 — state_count (int): Размер истории состояний (буфер). Ограничение: должно быть > 10.
 — error_count (int): Порог ошибок для блокировки. Ограничение: должно быть < 10.
 — network_errors (list[type[Exception]]): Список исключений, которые считаются ошибкой сети (например, [ConnectionError, TimeoutError]).
 — sleep_time_sec (int): Время охлаждения в секундах после ошибки.

## Домашнее задание №1
### Шаг 1. Каркас декоратора и сохранение метаданных
Задача: Реализовать базовую структуру декоратора-фабрики.
1. Создай функцию-декоратор circuit_breaker, которая принимает указанные выше аргументы.
2. Внутри создай функцию-обертку (wrapper), которая принимает аргументы декорируемой функции. Эти аргументы в декораторе д.б. уже на первом шаге:
 — state_count (int): Размер истории состояний (буфер). Ограничение: должно быть > 10.
 — error_count (int): Порог ошибок для блокировки. Ограничение: должно быть < 10.
 — network_errors (list[type[Exception]]): Список исключений, которые считаются ошибкой сети (например, [ConnectionError, TimeoutError])
3. Обеспечь сохранение метаданных оригинальной функции (имя, докстринг) с помощью functools.wraps.
4. Реализуй простую логику: обертка должна просто вызывать оригинальную функцию и возвращать её результат.

### Шаг 2. История состояний и отлов исключений
Задача: Научить декоратор помнить историю вызовов и различать успех/ошибку.
1. В замыкании декоратора создай структуру данных для хранения истории (рекомендуется collections.deque с ограничением maxlen).
2. При вызове функции фиксируй результат:
 — Если функция отработала успешно -> запиши в историю True (или объект состояния с временем).
 — Если возникло исключение из списка network_errors -> запиши False.
 — Если возникло иное исключение -> пробрось его дальше, не записывая в историю (это баг кода, а не сети).
3. История не должна расти бесконечно, она хранит только последние state_count записей.

### Шаг 3. Логика Circuit Breaker
Задача: Реализовать блокировку вызовов при частых ошибках.
1. Перед вызовом оригинальной функции анализируй историю.
2. Если последние error_count записей в истории являются ошибками (False):
— Не вызывай оригинальную функцию. 
— Выброси кастомное исключение NotAliveError (создай класс исключения отдельно).
3. Если порог не достигнут - функция выполняется в штатном режиме.

### Шаг 4. Охлаждение и валидация аргументов
Задача: Добавь задержку при нестабильности и проверь входные данные.
1. Валидация:
 — При инициализации декоратора проверь аргументы. Если state_count <= 10 или error_count >= 10 (или другие нелогичные комбинации), выброси ValueError.
2. Задержка (Backoff):
 — Если последнее записанное состояние в истории – ошибка (False), то перед попыткой нового вызова программа должна уснуть на sleep_time_sec секунд.
 — Примечание: Задержка применяется только если цепь не разомкнута (проверка из Шага 3 не сработала).
